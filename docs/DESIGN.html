통합 에이전트-디스코드 봇 설계 · 승인 대기

봇 하나로 Claude Code와 Codex를 채널마다 골라 쓰는 자가호스팅 디스코드 봇. 여러 서버 지원 · 역할 기반 권한 · 3계층 설정 · DB 없음(JSON) · 새 백엔드/기능 확장 쉬움.

한마디로: 내 PC에서 봇 하나를 돌리면, 디스코드 채널마다 "이 채널은 Claude, 저 채널은 Codex"처럼 골라서 코딩 에이전트를 쓸 수 있게 만드는 프로그램입니다. 참고한 두 프로젝트(A4D·CDC)의 약점을 고쳐서 더 안전하고 편하게 만드는 게 핵심 목표예요.

0. 레퍼런스를 어떻게 참고했나

한마디로: 두 참고 프로젝트에서 "연결하는 코드"와 "어떤 기능을 넣을지 아이디어"만 빌려오고, 구조(설계)는 이 프로젝트에 맞게 새로 짭니다. 레퍼런스를 그대로 따라 하지 않아요.
  • 연결 코드 — Claude Code는 @anthropic-ai/claude-agent-sdk의 query()를 프로세스 안에서 호출(근거 A4D/src/sessions/sessionManager.ts:76). Codex는 codex exec --json을 실행해 출력(JSON)을 읽고, ~/.codex에서 과거 세션을 찾음(근거 CDC/.../codexRunner.ts, parser.ts:96).
  • 기능 아이디어 — 툴 스레드, 권한 모드 선택, 스트리밍 임베드, 폴더 브라우저, 파일 첨부, 트랜스크립트, 역할 허용목록, 명령 안전정책, 사용량 패널 등 "어떤 걸 넣을지".

그 외 모듈 경계·모드 추상화·설정 구조·인증 위치는 전부 새로 설계했습니다.

1. 목표 / 안 할 것 / 보안

한마디로: "봇 1개 · 여러 서버 · 채널마다 백엔드 선택"이 핵심 모델이고, 두 레퍼런스보다 안전하고 튼튼하게 만드는 게 목표. DB·다중PC·SaaS는 안 합니다.

목표

  • 두 모드, 한 프로그램 — 채널마다 Claude Code 모드 또는 Codex 모드. 각 모드는 자기에게 맞는 화면을 갖고, 억지로 하나로 뭉개지 않음.
  • 봇 1개 · 여러 서버 · 채널별 백엔드 — 디스코드 봇 하나(토큰 1개)가 여러 서버에 참여하고, 채널 만들 때 Claude/Codex를 고름(A4D 모델 고르듯). 상태는 서버ID+채널ID로 구분.
  • 두 레퍼런스보다 낫게 — 약점 개선(§3)이 곁다리가 아니라 핵심 가치.
  • 역할 3단계 + 프로젝트별 접근제어 — admin(관리) / execute(실행) / read-only(보기)로 나눠, 슬래시 명령과 일반 메시지에 똑같이 적용.
  • 권한은 "모드 선택"으로 — 세션마다 권한 모드(일반/수정자동/전부자동/플랜)를 고름. 전역/서버/프로젝트 기본값 + 세션 중 /mode로 변경.
  • 사용량 표시는 Claude 전용 — 5시간·주간 한도 + 컨텍스트 사용량. Codex는 CLI 제약으로 미지원(정직하게 표기).
  • 확장성 — 새 백엔드(예: Gemini) 추가, 새 기능 추가가 둘 다 깔끔.
  • 자가호스팅 · JSON 상태 · npx 설치 — 내 PC에서 프로세스 하나로, 설정은 ~/.<이름>/*.json.

안 할 것 (명시)

  • DB 없음 — Prisma·자체 SQLite 안 씀. (Codex 세션 DB는 번들 리더로 읽기만.)
  • 다중 컴퓨터(Hub) 없음 — control-api·에이전트 WebSocket 없음. CDC의 가장 위험한 보안면 제거.
  • 여러 토큰/전용봇 없음 — "봇 1개, 채널별 백엔드"로 확정. 전용 Claude봇/Codex봇 분리는 미래 확장으로만.
  • 클라우드/SaaS 없음 · 최소공통으로 뭉개기 없음(Claude의 풍부한 UX를 Codex에 맞춰 깎지 않음).
⚠ 보안(꼭 읽기): 이 봇은 내 PC에서, 내 자격증명·파일 접근 권한으로 에이전트를 돌립니다. 그래서 봇이 들어간 어느 서버든 허용된 사용자는 내 PC에서 코드를 실행할 수 있어요. 봇을 신뢰할 수 없는 서버에 넣는 건 그 서버 사람들에게 셸 접근을 주는 것과 같습니다. → 서버·프로젝트별 역할 인증이 필수(기본 차단). 이 제약이 인증 모델·폴더 차단·기본 권한 모드·정지 버튼·감사 로그의 근거입니다.

3. 두 레퍼런스 단점 → 개선

한마디로: A4D는 "인증이 없고 재시작하면 세션이 날아가고 파일 유출 위험"이 있고, CDC는 "인증 없는 원격 실행 구멍(치명적)·조용한 실패"가 있습니다. 이걸 전부 고치는 게 이 프로젝트의 존재 이유예요.

대표 약점 3개 — 고치기 전(빨강) → 후(초록)

❌ CDC: 인증 없는 원격 실행 치명적

Hub의 WebSocket이 아무 토큰 없이 연결을 수락 → 같은 네트워크의 누구나 명령 실행 가능.

✅ 개선

Hub 자체를 완전히 삭제(다중PC 비목표). 네트워크 리스너·에이전트 등록이 아예 없어져 공격면 소멸.

❌ A4D: 인증이 아예 없음 높음

채널에 있는 사람은 누구나 Claude에 명령 가능. 역할 구분 없음.

✅ 개선

모든 입력 앞에 역할 인증 게이트(기본 차단). 메시지·슬래시 명령에 똑같이 적용.

❌ A4D: 재시작하면 세션 유실 높음

세션이 메모리에만 있어 봇/PC 재시작 시 전부 사라짐.

✅ 개선

부팅 시 자동 재개 — state.json을 읽어 채널마다 세션을 다시 연결.

4. 폴더 구조

한마디로: 모노레포(CDC)도, 평평한 구조(A4D)도 아닌 단일 패키지 + 관심사별 폴더. 확장성은 "폴더 쪼개기"가 아니라 인터페이스 경계에서 나옵니다.

③ 계층 의존 방향 (한 방향만!)

discord/화면(렌더러)
→
core/공용 계약·로직
(아무것도 의존 안 함)
←

5. 핵심 계약(모드 인터페이스)

한마디로: 모든 백엔드는 같은 "표준 플러그"(AgentMode)를 구현합니다. 각 모드는 "내가 뭘 할 수 있는지"(Capabilities=기능 체크리스트)를 선언하고, 공통 이벤트 언어(AgentEvent)로 말합니다.
  • AgentMode = 새 백엔드가 구현하는 단 하나의 표준 인터페이스(콘센트 규격). 시작/재개/전송을 정의.
  • Capabilities = "이 모드는 스트리밍 되나? 권한 버튼 되나? 사용량 패널 되나?" 체크리스트. 화면은 체크된 기능만 그림.
  • AgentEvent = 텍스트/생각/툴사용/권한요청/진행/결과/컨텍스트/에러를 담는 공통 이벤트. 어느 백엔드든 이 언어로 번역해 보냄.

  • ❌ 툴별 스레드 / 권한 버튼
  • ❌ 사용량/한도 패널(CLI 제약)
  • 권한은 Codex 승인·샌드박스 모드로 대체

6. 모드별 화면(UX)

한마디로: 화면은 "이 모드가 지원하는 기능만" 그립니다. Claude는 풀 화면, Codex는 진행상황·결과 위주로 간결하게(정직하게 덜 보여줌).

🤖 Claude 화면

  • 실시간으로 갱신되는 스트리밍 임베드(텍스트/생각)
  • 툴 실행마다 스레드 생성 + 결과 회신
  • 일반 모드에선 도구 실행 전 Allow/Deny 버튼(=그게 "확인")
  • 파일 수정 diff, 사용량 패널, 파일 첨부/다운로드(폴더 밖 차단)

⚙️ Codex 화면

  • 진행상황 트랜스크립트(예: "파일 수정 중")+ 최종 결과 메시지
  • 스트리밍·스레드·권한버튼·사용량 패널 없음
  • 상태 임베드에 "사용량/한도 미지원(Codex CLI 제약)" 한 줄
  • 권한: 권한 모드 → Codex 샌드박스/승인 플래그 + 명령정책 사전확인

7. 권한 · 인증 · 정책

한마디로: "누가 쓸 수 있나"(역할 3티어) + "봇이 어디까지 해도 되나"(권한 모드/프로필) + "위험 명령 거르기"(명령 정책)를 코어에서 모드 실행 전에 처리합니다.

④ 한 번의 대화가 처리되는 흐름

사용자 메시지
→
역할 인증허용? 아니면 차단
→
순서 대기열유실 없이 큐잉
→
권한 모드/정책위험하면 확인
→

인증 — 역할 3티어 + 프로젝트별 접근제어

  • admin — 설정 변경, 전체 정지, 감사 접근 · execute — 세션 시작·명령 실행(운전석) · read-only — 보기만(시작·수정 불가)
  • 허용목록이 비면 전부 차단(fail-secure). 프로젝트마다 허용 역할/사용자를 더 좁힐 수 있음(교집합).

권한 모드 & 프로필 (A4D의 권한 모드 선택 참고)

위험 행동 통제의 주 수단은 "권한 모드 선택"입니다(별도 확인 버튼이 아니라). A4D가 세션 시작 시 default/acceptEdits/bypassPermissions/plan/dontAsk를 고르게 하는 UI를 그대로 참고.

권한 모드ClaudeCodex(매핑, 2단계 검증)
plan(플랜)읽기·계획만sandbox: read-only
default(일반)도구 실행 전 Allow/Deny 버튼approval: on-request
acceptEdits(수정 자동수락)파일 수정 자동 승인sandbox: workspace-write
bypassPermissions(전부 자동) ⚠전부 자동 승인danger-full-access / approval:never
  • 프로필 = 권한 모드 + 허용 도구 + 정책 티어를 묶은 프리셋. 예: 읽기전용 / 수정허용 / 자동. 프로젝트마다 지정, 전역/서버에서 상속.
  • 권한 모드는 계층 기본값 + 세션 중 /mode로 변경 가능. 명령 정책은 그 아래 안전망으로 항상 동작.

명령 정책 · Codex DB 리더 · 기타 안전장치

  • 명령 정책 — CDC의 분류기(따옴표 인식 토크나이저 등)를 이식하되, "모르는 명령은 확인 요구"(fail-secure)로 뒤집음.
  • Codex 세션 DB — 외부 sqlite3 대신 JS/WASM 리더 번들로 ~/.codex/state_*.sqlite를 직접 읽음(설치 불필요, C3 완전 해결). 실패 시 인덱스 전용 + "모르면 제외" 안전망.
  • 킬 스위치 — /stop(현재 세션), /stop-all(관리자, 전 서버 전부 정지).
  • 감사 로그 — 누가/언제/무엇을 audit/*.jsonl에 기록(+선택적으로 디스코드 채널 미러). DB 없이.
  • 폴더 차단(기본) — 모든 파일 경로는 작업 폴더 밖으로 못 나감(실경로 확인). 권한 모드와 무관하게 항상.
  • 훅 → 디스코드 알림 — 로컬 전용 엔드포인트로 에이전트 훅 이벤트("테스트 실패" 등)를 받아 채널에 알림.

7.4 사용량 / 한도 패널

한마디로: 사용량 표시는 Claude 전용입니다. Claude는 5시간·주간 한도 + 컨텍스트 사용량까지 보여줄 수 있고(구독 로그인 필요), Codex는 CLI가 한도를 안 줘서 미지원입니다.

⑧ 사용량/한도 지원 현황

🤖 Claude — 셋 다 됨

  • ✅ 5시간 한도 남은량 + 리셋 시각
  • ✅ 주간 한도 남은량(모델별)
  • ✅ 컨텍스트 사용량 (사용/최대 %)

단, 5시간·주간은 "남은 토큰 수"가 아니라 사용률(%). Anthropic의 비공식 OAuth 사용량 엔드포인트 사용 → Claude Pro/Max 구독 로그인 필요(API 키만이면 패널 자동 숨김).

⚙️ Codex — 미지원

  • ❌ 5시간 한도 — CLI가 항상 null 반환
  • ❌ 주간 한도 — 제공 안 됨
  • ❌ 컨텍스트 사용량 — 최대치를 안 줘서 근사치도 안 함

상태 임베드에 "사용량/한도 미지원(Codex CLI 제약)" 한 줄만 표기. 정직하게.

8. 설정 3계층

한마디로: 설정은 전역 → 서버 → 프로젝트 순으로 덮어씁니다. 가장 구체적인 값이 이깁니다. 프로젝트마다 백엔드·권한·모델을 다르게 둘 수 있어요.

⑦ 설정이 결정되는 순서

🌐 전역 기본값config.json — 토큰·기본 백엔드·기본 권한 등
↓ 덮어쓰기
🏠 서버별servers/<서버ID>.json — 이 서버만 다르게
↓ 덮어쓰기
📁 프로젝트별state.json 채널 바인딩 — 이 채널만 다르게 (최종 우선)

계층화되는 값: 기본 백엔드 · 권한 모드 · 권한 프로필 · 모델 · 허용 역할 · 자동허용 도구 · 한도. (허용목록은 좁은 계층에서 더 좁히기만 가능, 넓히진 못함.)

9. 세션 수명주기

한마디로: 채널 만들 때 마법사로 폴더·백엔드·모델·권한을 한 번에 고르고, 재시작해도 자동으로 이어지며, /stop으로 즉시 멈출 수 있습니다.
  1. 시작(채널 마법사) — /agent start → 폴더(또는 즐겨찾기) → 백엔드 → 모델 → 권한 모드/프로필을 한 흐름으로. 기본값은 계층에서 자동 채움.
  2. 턴 처리 — 인증 후 대기열에 넣고 순서대로 실행. 파일은 폴더 밖 차단. 완료 시 선택적으로 작성자 @멘션. 모든 턴 감사 기록.
  3. 모드/권한 전환 — /mode로 백엔드·권한 변경. 단, 백엔드를 바꾸면 세션 ID가 호환 안 돼 맥락이 안 넘어감 → 허용 + 경고(확정):
    ⚠️ Codex로 바꾸면 이 채널은 새 대화로 시작돼요. 이전 맥락은 안 넘어갑니다.
    권한 모드만 바꾸면 세션 유지, 다음 턴부터 적용.
  4. 재개 — 부팅 시 state.json을 읽어 자동 재연결(Claude는 SDK resume, Codex는 다음 턴에 exec resume). 필요시 /agent resume로 과거 세션 선택.
  5. 정지/보관 — /stop(현재), /stop-all(관리자, 전부). 채널 보관/삭제도 지원.

10. 확장 방법

한마디로: 새 백엔드나 새 기능을 추가해도 다른 부분을 안 건드립니다. core가 유일한 공용 의존이기 때문.

새 백엔드 추가 (예: Gemini)

  1. modes/gemini/ 만들어 AgentMode 구현(기능 선언)
  2. 연결 코드 + 이벤트 매퍼 작성
  3. 레지스트리에 한 줄 등록
  4. 끝. discord/auth/정책/상태 무수정

새 기능/명령 추가 (예: /agent stats)

  1. 명령 라우터에 정의+핸들러
  2. 필요하면 렌더러 하나 추가
  3. 인증은 자동(라우터가 먼저 검사)
  4. 영향 범위 = 명령 파일 + 렌더러 하나

13. 리스크 / 남은 결정

한마디로: 가장 큰 리스크는 "봇이 내 PC에서 코드를 실행한다"는 보안 특성과, Codex의 CLI 제약(한도·이벤트 스키마)입니다. 대부분 완화책이 있고, 두 개(모드 전환·sqlite3)는 이미 결정됐습니다.

다중 서버 = 넓은 공격면

봇이 든 어느 서버든 허용자가 내 PC에서 코드 실행 가능. → 기본 차단·역할 티어·프로젝트 ACL·폴더 차단·보수적 기본 권한·감사 로그로 완화. 잔여: 운영자 오설정(설치 경고로 안내).

모드 간 기능 차이는 본질적

Claude=지속 스트리밍+권한 노출, Codex=1회성+사후 트랜스크립트. 모드 모델로 수용하되, 상태 임베드/문서로 차이를 명시.

Codex 권한 + 플래그 검증

권한 모드를 Codex 승인/샌드박스 플래그에 매핑하지만 액션별 승인은 불가. → 2단계에서 실제 CLI 플래그 검증 + 명령정책 사전확인 + 정직한 문서.

비공식 OAuth 사용량 엔드포인트

Claude 5시간/주간 패널이 비공식 API에 의존 → 바뀌거나 막힐 수 있음. 구독 로그인 시만·캐시·백오프·문제 시 우아하게 숨김. (컨텍스트 사용량은 SDK라 영향 없음.)

Codex 한도 안 보임
